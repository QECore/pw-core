# Release Notes (v1.2.0)

This release introduces significant enhancements to test reporting, framework transparency, dynamic configuration support, and table validation features.

## 🌟 Release Highlights
- **[IntelliSense-Powered Dynamic Selectors](#14-create-flexible-dynamic-elements-on-the-fly-with-auto-complete-support)**: Generate multi-variable dynamic selectors on the fly while retaining full auto-complete in your editor to eliminate typos.
- **[Zero-Code Test Step Documentation](#15-automatically-document-test-actions-in-html-reports)**: Custom page actions are automatically wrapped and documented as readable test steps in HTML reports via prototype reflection.
- **[Frictionless Stack-Trace Cleanups](#16-point-test-failures-directly-to-the-exact-test-line-in-reports)**: Automatically filters framework files from report logs, pointing failure lines directly back to your test spec file.

## Released Tickets

### [#14 Create Flexible Dynamic Elements on the Fly with Auto-Complete Support](https://github.com/QECore/pw-core/issues/14)
* **Type**: Feature
* **Area**: `Page Configs`
* **Description**: Allow page elements to be defined using dynamic variables (e.g., `{status}{type}Chart`), while still ensuring the code editor suggests all possible combinations automatically (like `activeLineChart` or `inactiveBarChart`) to prevent typos.
* **Technical Example**:
  ```ts
  import { createPageConfig } from 'pw-core/page';

  export const config = createPageConfig({
    testIds: {
      "{status}{id}Chart": {
        status: ["active", "inactive"] as const,
        id: ["Line", "Bar"] as const,
        testId: "chart-{status}-{id}"
      }
    }
  });

  // Auto-complete in IDE now suggests:
  // "activeLineChart" | "activeBarChart" | "inactiveLineChart" | "inactiveBarChart"
  await page.verify("activeLineChart"); 
  ```

### [#15 Automatically Document Test Actions in HTML Reports](https://github.com/QECore/pw-core/issues/15)
* **Type**: Feature
* **Area**: `Page Objects`
* **Description**: Automatically record user-defined actions (such as logging in, searching, or checking out) as readable steps inside the Playwright HTML test report, so anyone looking at the reports can easily understand what actions the test performed.
* **Technical Example**:
  ```ts
  class CheckoutPage extends TypedPage<typeof config> {
    async fillBillingDetails(name: string, zip: string) {
      await this.fill("nameInput", name);
      await this.fill("zipInput", zip);
    }
  }

  // When called in a test:
  await checkoutPage.fillBillingDetails("John Doe", "90210");

  // The Playwright HTML report automatically registers a step named:
  // "⚡ Fill Billing Details - John Doe - 90210"
  ```

### [#16 Point Test Failures Directly to the Exact Test Line in Reports](https://github.com/QECore/pw-core/issues/16)
* **Type**: Feature
* **Area**: `Test Reports`
* **Description**: Filter out internal framework code files from failure reports. When a test fails, the report will highlight the exact line in your test script that failed, rather than showing complex code lines from inside the testing library.
* **Technical Example**:
  ```ts
  // e2e.test.ts
  await dashboard.verify("nonExistentElement"); // Fails here
  ```
  * **Before Filtering**: Error points to `node_modules/pw-core/dist/page/assertions/verify-helpers.js`.
  * **After Filtering**: Error highlights the exact caller line in your spec: `examples/src/tests/e2e.test.ts:L14`.

---

### [#17 Improve Error Reporting When Retrieving Grid Table Values](https://github.com/QECore/pw-core/issues/17)
* **Type**: Feature
* **Area**: `Test Reports`
* **Description**: Wrap table-reading operations (like checking a specific cell value or counting rows) in clean report steps, and ensure any table-related errors point directly back to the test file instead of internal package files.
* **Technical Example**:
  ```ts
  import { Table } from 'pw-core/component/table';

  const table = new Table(page.locator('.data-table'));

  // Select the cell value at row 0, column "status"
  const cellValue = await table.getCellValue(0, "status");
  expect(cellValue).toBe("Pending");
  ```
