# Release Notes (v1.1.0)

This release introduces the initial core package of the `pw-core` framework, featuring typed page objects, a page registry for automatic fixture setup, chained locators, direct verification assertions, and custom table mapping components.

## 🌟 Release Highlights
- **[Zero-Boilerplate Auto-Fixtures](#page-registry)**: Setup screens once and use them directly as test arguments—completely avoiding repetitive boilerplate setup.
- **[Fluent Direct Verifications](#readable-assertions)**: Write assertions directly on your page objects instead of wrapping elements inside verbose, nested check blocks.
- **[Dot-Notation Element Access](#chained-locators)**: Target nested elements inside containers easily using dot paths instead of writing long, nested selection chains.
- **[Instant HTML Table-to-Data Mapper](#reusable-components)**: Automatically maps visual HTML table grids into clean, queryable lists of structured objects.
- **[AI-Optimized Predictable Commands](#ai-friendly-automation)**: Unifies all screen interactions under one predictable interface, reducing token use and bugs for AI agents.

## Released Tickets

### [#7 Provide a Standard Template to Build and Organize Screen Objects](https://github.com/QECore/pw-core/issues/7)
* **Type**: Feature
* **Area**: `Page Objects`
* **Description**: Create a central template that represents a web page (Screen Object). This allows test writers to define all page element locations (like textboxes, buttons, and links) in one place so that tests stay clean and updates to the screen design only need to be changed in a single file instead of across hundreds of tests.

### [#8 Automatically Inject Screens Into Tests to Avoid Repetitive Setup](https://github.com/QECore/pw-core/issues/8)
* **Type**: Feature
* **Area**: `Test Fixtures`
* **Description**: Allow screens to be registered once and automatically loaded into tests. Test writers no longer have to write repetitive setup code (like importing and initializing page objects manually) at the beginning of every single test file.

### [#9 Check Screen Elements Directly with Simple Verification Commands](https://github.com/QECore/pw-core/issues/9)
* **Type**: Feature
* **Area**: `Test Assertions`
* **Description**: Simplify the syntax for verifying elements on the screen. Instead of wrapping element selectors in verbose code to check if they are visible, active, or disabled, allow direct validation on the element itself in a single, simple, readable line of code.

### [#10 Find Nested Elements on a Page Using Simple Dot Notation](https://github.com/QECore/pw-core/issues/10)
* **Type**: Feature
* **Area**: `Element Locators`
* **Description**: Provide a way to select elements nested deep inside other sections (like a form inside a popup modal) by using a dot-connected path (e.g., "modal.form.submit"), rather than writing complex, multi-line nested code that is hard to maintain.

### [#11 Read and Interact with Grid and Table Data as Structured Lists](https://github.com/QECore/pw-core/issues/11)
* **Type**: Feature
* **Area**: `Table Component`
* **Description**: Introduce a reusable table component that translates visual HTML grid rows and columns into clean lists of data. This makes it easy to read, find, or check table values in automated tests without custom web elements parsing.

### [#12 Standardize Screen Actions to Make Test Creation Simpler for AI Assistants](https://github.com/QECore/pw-core/issues/12)
* **Type**: Feature
* **Area**: `Test Scripts`
* **Description**: Standardize all screen commands (e.g., clicking, filling text, validating) into a uniform structure. This allows AI assistants and test generation engines to write, read, and maintain tests faster and with far fewer errors.

### [#13 Save and Load Web Browser Memory State During Tests](https://github.com/QECore/pw-core/issues/13)
* **Type**: Feature
* **Area**: `Test Scripts`
* **Description**: Add quick utilities to write and read from the browser's temporary memory storage (LocalStorage and SessionStorage). This is useful for pre-logging a user in or checking that user settings are correctly saved.

# Page Registry

Not every page requires a custom Page Object. Many pages only contain URLs, testIds, and selectors. For those pages, use `createPageRegistry`.

```ts
import { createPageRegistry } from "pw-core/page";

export const test = createPageRegistry({
  loginPage: {
    url: "/login",

    testIds: {
      username: "username-input",
      password: "password-input",
      submit: "login-button",
    },
  },
});
```

## Automatic Fixtures

Pages registered in `createPageRegistry` immediately become fixtures in your test runner:

```ts
test("login", async ({ loginPage }) => {
  await loginPage.fill("username", "admin");
  await loginPage.fill("password", "password");
  await loginPage.click("submit");
});
```

No Page Object class or custom fixture setup is required!

# TypedPage

`TypedPage` is the foundation of `pw-core`. It transforms page configurations into fully typed automation objects.

```ts
import { TypedPage, createPageConfig } from 'pw-core/page';

export const config = createPageConfig({
  url: "/login",
  testIds: {
    username: "username-input",
    submitBtn: "login-button",
  },
});

class LoginPage extends TypedPage<typeof config> {
  async login(user: string, pass: string) {
    await this.fill("username", user);
    await this.click("submitBtn");
  }
}
```

# Extending Registry Pages

As pages become more complex, you can extend them with custom classes:

```ts
class LoginPage extends test.pages.loginPage {
  async login() {
    await this.fill("username", "admin");
    await this.fill("password", "password");
    await this.click("submit");
  }
}
```

Update the fixture:

```ts
export const extendedTest = test.extend({
  loginPage: LoginPage,
});
```

Usage:

```ts
await loginPage.login();
```

Start simple and extend only when needed.

# Readable Assertions

Traditional Playwright assertions require wrapping elements in `expect(locator)` blocks:
```ts
await expect(page.getByTestId("login-button")).toBeVisible();
```

With `pw-core`, assertions are chainable and directly accessible:
```ts
await login.verify("loginBtn");
```

Custom assertions remain available:
```ts
await login.verify("loginBtn").toBeEnabled();
await login.verify("errorMessage").toContainText("Invalid");
```

---

# Chained Locators

Complex nested locators stay highly readable regardless of depth.

Traditional:
```ts
await page
  .locator('[data-testid="modal"]')
  .locator('[data-testid="form"]')
  .locator('[data-testid="submit"]')
  .click();
```

pw-core:
```ts
await page.click("modal.form.submit");
```

# Reusable Components

With `pw-core`, components become first-class citizens. Define once and reuse across different pages:

```ts
export class TransactionsPage extends TypedPage<typeof config> {
  constructor(page: Page) {
    super(page, config);
    this.transactionsTable = new Table<Transaction>(tableLocator);
  }
}
```

# AI-Friendly Automation

`pw-core` standardizes and simplifies the interaction pattern to make it highly friendly for AI agents.

Instead of writing custom combinations of `page.locator()`, `page.getByRole()`, `page.getByTestId()`, all actions follow a standardized, predictable API:

```ts
page.click(...)
page.fill(...)
page.verify(...)
page.locator(...)
```

Benefits:
- Reduced context size & token usage
- Better and more predictable AI-generated tests
- Faster onboarding and easier code reviews

# When To Use TypedPage vs Page Registry

### Use TypedPage when:
- The page contains complex workflows
- The page contains reusable business logic
- Multiple tests interact with the page

### Use Page Registry when:
- Pages only contain locators
- No custom page methods are needed
- You want minimal boilerplate and automatic fixtures

# Traditional Playwright vs pw-core Comparison

## Traditional Approach

```ts
class LoginPage {
  constructor(private page: Page) {}

  username = this.page.getByTestId("username-input");
  password = this.page.getByTestId("password-input");
  loginButton = this.page.getByTestId("login-button");
  errorMessage = this.page.getByTestId("login-error");
}
```

Usage:

```ts
await loginPage.username.fill(user);
await loginPage.password.fill(pass);
await loginPage.loginButton.click();

await expect(loginPage.errorMessage).toBeVisible();
```

---

## pw-core Approach

```ts
const config = createPageConfig({
  url: "/login",

  testIds: {
    username: "username-input",
    password: "password-input",
    loginBtn: "login-button",
    loginError: "login-error",
  },
});

class LoginPage extends TypedPage<typeof config> {
  constructor(page: Page) {
    super(page, config);
  }
}
```

Usage:

```ts
await loginPage.fill("username", "admin");
await loginPage.fill("password", "password");
await loginPage.click("loginBtn");
await login.verify("loginError");
```
